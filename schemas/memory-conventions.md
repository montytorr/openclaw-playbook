# Memory Conventions — File Formats & Standards

This document specifies the file formats, naming conventions, and schemas used by the memory system.

## Daily Note Files

### Naming Convention

```
memory/YYYY-MM-DD.md
```

Examples:
- `memory/2026-04-03.md`
- `memory/2026-03-15.md`

One file per day. Created when the agent first writes to it (no need to pre-create).

### File Structure

```markdown
# YYYY-MM-DD

## HH:MM — Context Label
- Bullet point observations
- Decisions made and why
- Things learned or discovered

## HH:MM — Another Context Label
- More observations
- Changes made to systems
- Errors encountered and fixes applied
```

**Rules:**
- Use 24-hour time format (HH:MM)
- Group entries by time and context (what was happening)
- Each entry should stand alone — a future reader shouldn't need the full conversation to understand it
- Include the "why" for decisions, not just the "what"

### Content Categories

| Category | What to Capture | Example |
|----------|----------------|---------|
| Decision | What was decided and why | "Chose PostgreSQL over MySQL for the new service because of JSON support" |
| Discovery | Something learned about a system or process | "Docker bridge DNS fails after daemon restart — need explicit aliases" |
| Change | What was modified | "Updated Traefik config to add new route for dashboard" |
| Error | What went wrong and how it was fixed | "Dashboard 500 error caused by null reference in task render" |
| Context | Background info for future sessions | "Human is traveling next week — reduce proactive notifications" |
| Person | New info about someone | "Met with Alex — they're handling the API integration" |

### What NOT to Include

- Secrets, API keys, or credentials
- Routine read operations ("Read SOUL.md" — not useful)
- Trivial conversation that contains no decisions or context
- Duplicate information already in project files

## MEMORY.md Format

MEMORY.md is auto-generated by the observation extraction tool. Do not edit manually.

### Structure

```markdown
# MEMORY.md — Observation Index
<!-- Auto-generated by claude-mem. Do not edit manually. -->
<!-- Last updated: 2026-04-03T10:00:00Z -->
<!-- Observations: 47 of 1,234 total -->

## Recent Observations

| ID | Date | Category | Observation |
|----|------|----------|-------------|
| 1234 | 2026-04-03 | decision | Switched dashboard to read-only SQLite access |
| 1233 | 2026-04-03 | change | Added integrity-check cron job (every 6 hours) |
| 1230 | 2026-04-02 | error | Docker networking broke after update — fixed with explicit bridge config |
| 1225 | 2026-04-02 | discovery | claude-mem extraction misses entries without timestamps — always include HH:MM |
| ... | ... | ... | ... |

## Key Patterns

<!-- Auto-generated pattern summaries -->
- Docker issues cluster after daemon restarts (seen 3x in 2 weeks)
- Trading recommendations rejected when based on momentum during low volume
- Dashboard deployments need cache clear after config changes
```

### Generation Rules

The observation extraction tool should:
1. Parse all `memory/YYYY-MM-DD.md` files
2. Extract individual observations (each bullet point or logical unit)
3. Classify by category (decision, discovery, change, error, context, person)
4. Score by importance (recency + impact + frequency of related topics)
5. Store in SQLite with full metadata
6. Generate MEMORY.md with top ~50 observations sorted by relevance
7. Include pattern summaries when themes repeat across multiple days

## Observation Database Schema

```sql
CREATE TABLE IF NOT EXISTS observations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    category TEXT NOT NULL DEFAULT 'context'
        CHECK (category IN ('decision', 'discovery', 'change', 'error', 'context', 'person')),
    importance REAL NOT NULL DEFAULT 0.5
        CHECK (importance >= 0.0 AND importance <= 1.0),
    source_file TEXT NOT NULL,           -- e.g., "memory/2026-04-03.md"
    source_date TEXT NOT NULL,           -- YYYY-MM-DD
    extracted_at TEXT NOT NULL DEFAULT (datetime('now')),
    tags TEXT                            -- JSON array of tags, optional
);

CREATE INDEX IF NOT EXISTS idx_obs_date ON observations(source_date DESC);
CREATE INDEX IF NOT EXISTS idx_obs_category ON observations(category);
CREATE INDEX IF NOT EXISTS idx_obs_importance ON observations(importance DESC);

-- Full-text search index
CREATE VIRTUAL TABLE IF NOT EXISTS observations_fts USING fts5(
    content,
    content=observations,
    content_rowid=id
);
```

## Heartbeat State

### File

```
memory/heartbeat-state.json
```

### Schema

```json
{
  "lastChecks": {
    "email": 1712131200,
    "calendar": 1712127600,
    "weather": null,
    "system_health": 1712120400,
    "tasks": 1712131200
  },
  "lastHeartbeat": 1712131200,
  "checksToday": 4,
  "lastDate": "2026-04-03"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `lastChecks` | Object | Unix timestamps of last check per category. `null` = never checked. |
| `lastHeartbeat` | Number | Unix timestamp of last heartbeat |
| `checksToday` | Number | How many heartbeat checks today (resets daily) |
| `lastDate` | String | Date of last check (for daily reset detection) |

### Behavior

On each heartbeat:
1. Read `heartbeat-state.json`
2. Determine which checks are due (>30 min since last check, or never checked)
3. Perform due checks
4. Update timestamps and `checksToday`
5. Write updated state back to file

## Retention & Archival

### Daily Notes

| Age | Location | Action |
|-----|----------|--------|
| 0-30 days | `memory/YYYY-MM-DD.md` | Active — loaded in session startup |
| 31-90 days | `memory/YYYY-MM-DD.md` | Still accessible but not loaded by default |
| 91+ days | `memory/archive/YYYY/MM/` | Archived — move to subdirectory |

### Archival Script Pattern

```bash
# Move old daily notes to archive
find memory/ -maxdepth 1 -name "20*.md" -mtime +90 | while read f; do
    date_part=$(basename "$f" .md)
    year=${date_part:0:4}
    month=${date_part:5:2}
    mkdir -p "memory/archive/$year/$month"
    mv "$f" "memory/archive/$year/$month/"
done
```

### Observation Database

The SQLite database retains all observations indefinitely. Old observations naturally fall out of MEMORY.md (due to relevance scoring) but remain searchable via `mem-search`.

### MEMORY.md

Regenerated on each session start. No archival needed — it's always fresh.

## File Size Guidelines

| File | Target Size | Why |
|------|-------------|-----|
| Daily note | 500-2000 chars | Loaded in every session — keep lean |
| MEMORY.md | 3000-8000 chars | Loaded in main sessions — balance context vs token cost |
| heartbeat-state.json | <500 chars | Read/written on every heartbeat |

If daily notes consistently exceed 2000 characters, you're logging too much detail. Capture decisions and context, not play-by-play transcripts.
